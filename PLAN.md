## 新たに確認された不具合

コードベースを改めてスキャンし、`src/game/regression.test.ts` / `src/state/regression.test.ts` で再現確認を行った結果、以下の不具合が確認された。

1. **Pinfu の待ち条件未チェック**（`src/game/yaku.ts`）
   - 4順子 + 非役牌雀頭の条件だけで Pinfu を判定しており、両面待ち条件のチェックがない
   - 単騎待ち・辺張待ち・嵌張待ちでも Pinfu と誤判定される

2. **副露あり手の聴牌判定が `findTenpaiTiles(allTiles)` で誤判定される**（`src/state/GameState.ts`）
   - `handleExhaustiveDraw`（流局時の聴牌判定）で、`hand + meld.tiles` をまとめて `findTenpaiTiles` に渡している
   - `findTenpaiTiles` は副露面子を固定せずに牌を分解するため、実際には聴牌していない副露手が聴牌と誤判定される
   - `DECLARE_RIICHI` と `processAiTurn` のリーチ判定でも `findTenpaiTiles(testHand)` を使用しており、同様の問題がある
   - 既存の `findWaits` は副露を固定して正しく待ちを計算しているが、これらの箇所では使われていない

3. **和了時の特殊フラグが `false` 固定**（`src/state/GameState.ts`）
   - `RON` / `TSUMO` ケースで `fullScore` に `isDoubleRiichi` / `isIppatsu` / `isHaitei` / `isHoutei` / `isRinshan` / `isChankan` を `false` 固定で渡している
   - `src/game/yaku.ts` 側はこれらのフラグに基づいて役を付ける実装になっているが、GameState から正しい値が渡されていない
   - ダブルリーチ・一発・海底・河底・嶺上開花・槍槓が実質的に発生しない

4. **加槓後の槍槓（他者からのロン）未実装**（`src/state/GameState.ts`）
   - `KAKAN` アクション後、加槓した牌を他者がロンできる機会がない
   - 現在は `KAKAN` 後に即座に `pendingRinshan: true` を設定して嶺上ツモへ進むのみ
   - 本来は加槓した牌を対象に他者にロン機会を与え、ロンがなければ嶺上ツモへ進む必要がある

## 作業手順

### Phase 7: Pinfu の待ち条件チェック

**目的**: Pinfu を両面待ちのみに限定し、単騎・辺張・嵌張待ちで誤判定されないようにする。

**内容**:

- `src/game/yaku.ts` の Pinfu 判定で、`decomposeStandardHand` の結果から和了牌を含む順子を特定する
- 和了牌がその順子の両端（両面待ち）にある場合のみ Pinfu とする
- 単騎待ち・辺張待ち・嵌張待ちの場合は Pinfu を付けない

**受け入れ条件**:

- [ ] 単騎待ちで Pinfu が付かない
- [ ] 辺張待ちで Pinfu が付かない
- [ ] 嵌張待ちで Pinfu が付かない
- [ ] 両面待ちで Pinfu が付く
- [ ] 既存の Pinfu 関連テストが全て通る

**作成・変更予定ファイル**:

- `src/game/yaku.ts`
- `src/game/yaku.test.ts`

### Phase 8: 副露あり手の聴牌判定厳密化

**目的**: 副露済み面子を固定グループとして扱い、聴牌判定を厳密にする。

**内容**:

- `handleExhaustiveDraw` での聴牌判定を `findWaits` に置き換える
  - `findWaits(closedTiles, melds)` は既存の `isCompleteHand` を使い、副露を固定して待ちを計算する
- `DECLARE_RIICHI` と `processAiTurn` のリーチ判定も `findWaits` を使うように統一する
- リーチ時の待ち表示文字列は `findWaits` の結果から生成する

**受け入れ条件**:

- [ ] 副露あり手で実際に聴牌していない場合、流局時に不聴と判定される
- [ ] 副露あり手で実際に聴牌している場合、流局時に聴牌と判定される
- [ ] 副露あり手でリーチできない場合はリーチ宣言が拒否される
- [ ] 副露あり手でリーチできる場合はリーチ宣言が通る
- [ ] 新規テストが追加され `pnpm test` で通る

**作成・変更予定ファイル**:

- `src/state/GameState.ts`
- `src/state/GameState.test.ts`

### Phase 9: 和了時フラグの正確な計測と渡し

**目的**: ダブルリーチ・一発・海底・河底・嶺上開花・槍槓の各フラグを正しく判定し、`fullScore` に渡す。

**内容**:

- `GameState` に各プレイヤーの「リーチ宣言時の巡目」や「リーチ後一発機会フラグ」などを追加する
  - ダブルリーチ：配牌後、最初の自分のターン（第1巡目）以内にリーチした場合
  - 一発：リーチ宣言後、自分の次のツモまでに和了した場合
  - 海底：山の最後の牌をツモって和了
  - 河底：山が尽きた後の捨て牌でロン
  - 嶺上開花：嶺上ツモで和了
  - 槍槓：加槓した牌でロン
- `RON` / `TSUMO` ケースでこれらのフラグを `fullScore` に正しく渡す
- 各フラグは和了時にリセットまたは更新する

**受け入れ条件**:

- [ ] ダブルリーチが正しく判定され 2翻となる
- [ ] 一発が正しく判定され 1翻となる
- [ ] 海底ツモが正しく判定される
- [ ] 河底ロンが正しく判定される
- [ ] 嶺上開花が正しく判定される
- [ ] 槍槓が正しく判定される
- [ ] 新規テストが追加され `pnpm test` で通る

**作成・変更予定ファイル**:

- `src/state/GameState.ts`
- `src/state/GameState.test.ts`

### Phase 10: 加槓後の槍槓実装

**目的**: 加槓した牌を他者がロンできるようにし、ロンがなければ嶺上ツモへ進む。

**内容**:

- `KAKAN` アクション後、加槓した牌を `lastDiscard` に設定し、`collectClaims` で他者のロン候補を収集する
  - ロン候補がある場合は `phase: 'claiming'` にして他者にロン機会を与える
  - ロンが実行された場合は `isChankan: true` で和了計算
  - ロン候補がない場合、または全員が見逃した場合は嶺上ツモへ進む
- 槍槓ロン後は通常のロン処理と同じく点数移動・局終了を行う

**受け入れ条件**:

- [ ] 加槓した牌で他者がロンできる
- [ ] 槍槓ロンの和了に `isChankan: true` が渡され、槍槓役が付く
- [ ] 槍槓がない場合は嶺上ツモへ正常に進む
- [ ] 新規テストが追加され `pnpm test` で通る

**作成・変更予定ファイル**:

- `src/state/GameState.ts`
- `src/state/GameState.test.ts`
