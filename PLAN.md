# 東風戦完走に向けた作業手順

## 目的

`CONTEXT.md` で定義された「東風戦を完走できる」状態に到達する。

## 完了の定義

- `pnpm exec tsc --noEmit` がエラーなく通る
- `pnpm test` が全て通る
- 東風戦の統合テストが追加され、対戦が開始から終了まで自動で完走することを検証する
- 発見された既存のルール不具合が修正されている

## 現状

- TypeScript 型検査：パス
- ユニットテスト：6ファイル / 92テスト パス
- TODO/FIXME：なし
- 発見された不具合：下記 5件

## 確認された不具合

1. **リンシャンツモ未実装**（`src/state/GameState.ts`）
   - カン（暗槓・大明槓・加槓）後に王牌から嶺上牌を引かない
   - カン後の手牌枚数がずれ、嶺上開花が判定されない
2. **副露手のロン判定で副露牌を手牌と混在**（`src/state/GameState.ts`）
   - `collectClaims` と `isFuritenFromOwnDiscards` で副露済み面子の牌も分解対象にしている
3. **雀頭の符が重複しない**（`src/game/scoring.ts`）
   - 場風と自風が一致する場合（例：親の東対子）に本来の +4符 ではなく +2符 になる
4. **三暗槓・四暗槕で暗槓を刻子として数えない**（`src/game/yaku.ts`）
   - 暗槓を含む手で三暗槕・四暗槕が正しく判定されない
5. **UI のリーチボタンがテンパイを検証しない**（`src/ui/App.tsx`）
   - 表示上の問題。Reducer 側では弾くため致命的ではない

## 作業手順

### Phase 0: 現状把握（完了済み）

- [x] ビルド・テスト状況の確認
- [x] コードベースの網羅的なギャップ分析
- [x] 優先順位の決定

### Phase 1: 統合テストの追加

**目的**: 東風戦の完走を再現可能な形で保証する。

**内容**:

- `buildWall` に任意の乱数関数を注入できるようリファクタリングする
  - デフォルトは `Math.random()` のまま
  - テストでは固定シードまたは固定牌山を渡せるようにする
- 東風戦の全自動シミュレーションテストを追加する
  - AI 同士で東 1 局から終了まで自動再生
  - 牌山は固定し、再現性を持たせる

**受け入れ条件**:

- [ ] 統合テストが `pnpm test` で通る
- [ ] 対戦開始から終了まで例外が発生しない
- [ ] 全プレイヤーの持点 + 供託リーチ棒 × 1000 = 100,000 が常に成り立つ
- [ ] 局が東 1 → 東 2 → 東 3 → 東 4 → 終了 と正しく進む
- [ ] 対戦終了時に `phase === 'ended'` かつ `finalRanking` が設定される

**作成・変更予定ファイル**:

- `src/game/tiles.ts`（`buildWall` の乱数注入対応）
- `src/state/GameState.test.ts` または `src/state/GameState.integration.test.ts`

### Phase 2: リンシャンツモの実装

**目的**: カン後の正しい局進行と牌数整合性を保つ。

**内容**:

- 暗槓・大明槓・加槕のいずれの後も、王牌の末尾から嶺上牌を 1 枚引く
- カンドラを 1 枚めくる（既存の `revealKanDora` を利用）
- スコア計算に「嶺上ツモかどうか」を示すフラグを渡せるように準備する
  - 嶺上開花（Rinshankai）の役判定自体は Phase 5 以降で追加

**受け入れ条件**:

- [ ] カン後に嶺上牌を引き、手牌が正しい枚数に戻る
- [ ] カン後にカンドラが 1 枚増える
- [ ] 統合テストにカン後の局進行が含まれるよう拡張し、テストが通る

**作成・変更予定ファイル**:

- `src/state/GameState.ts`（Reducer の ANKAN / DAIMINKAN / KAKAN 処理）
- `src/state/GameState.test.ts`

### Phase 3: 雀頭符の重複修正

**目的**: 場風と自風が一致する雀頭の符を正しく +4符 とする。

**内容**:

- `src/game/scoring.ts` の雀頭符計算を修正し、場風と自風が一致しても両方加算する

**受け入れ条件**:

- [ ] 親が東対子を雀頭に持つ手で `calculateFu` が雀頭部分で 4符 を返す
- [ ] 子が東対子を持つ場合は 2符
- [ ] 親が南対子を持つ場合は 2符
- [ ] 三元牌の雀頭は 2符
- [ ] 新規テストが追加され `pnpm test` で通る

**作成・変更予定ファイル**:

- `src/game/scoring.ts`
- `src/game/scoring.test.ts`

### Phase 4: 副露手のロン・フリテン判定の厳密化

**目的**: 副露済み面子を固定グループとして扱い、ロン・フリテン判定を厳密にする。

**内容**:

- `isFuritenFromOwnDiscards` では `player.hand` のみで聴牌牌を計算する
- `collectClaims` のロン判定では、`detectYaku` または `decomposeStandardHand` を使い、副露を含めた和了形を判定する

**受け入れ条件**:

- [ ] 副露手でロン可能な手が正しくロン候補になる
- [ ] 副露手でフリテンになる手が正しくロンできない
- [ ] 新規テストが追加され `pnpm test` で通る

**作成・変更予定ファイル**:

- `src/state/GameState.ts`
- `src/state/GameState.test.ts`

### Phase 5: 暗槓を含む三暗槕・四暗槕判定

**目的**: 暗槕を concealed group として正しく数える。

**内容**:

- 三暗槕：刻子 + 暗槕をまとめて concealed group として数える
- 四暗槕：暗槕のみの副露は門前扱いとし、大明槓・加槕は門前ではない
- 嶺上開花フラグがあれば、このフェーズで Rinshankai 役判定も追加可能

**受け入れ条件**:

- [ ] 暗槕を含む三暗槕が検出される
- [ ] 暗槕のみの副露を持つ四暗槕が検出される
- [ ] 大明槕・加槕を含む手が四暗槕として誤検出されない
- [ ] 新規テストが追加され `pnpm test` で通る

**作成・変更予定ファイル**:

- `src/game/yaku.ts`
- `src/game/yaku.test.ts`

### Phase 6: 最終検証

**内容**:

- `pnpm exec tsc --noEmit`
- `pnpm test`
- 手動で 1 局プレイし、東風戦が終了画面まで到達することを確認

**受け入れ条件**:

- [ ] TypeScript 型検査が通る
- [ ] 全テストが通る
- [ ] 手動プレイで東風戦が完走する

## 備考

- ADR は作成しない。今回の決定は reversible で、コードベースへの影響が限定的。
- `CONTEXT.md` は新たに解決した用語がないため更新しない。
- 最終順位の同点処理については、現状 `rankPlayers` がプレイヤーインデックス順でソートしている。起家は常にプレイヤー 0 として初期化されるため、現状の実装で「起家に近い席を上位」とするルールは満たされている。起家が可変になった場合は別途対応が必要。
